# 方案评估报告：OpenCode Shell 自动修复插件

## 评估概要

| 项目 | 内容 |
|------|------|
| 评估目标 | 对"Shell 自动修复插件"简化版设计方案进行全面技术审查 |
| 方法 | 对照 `@opencode-ai/plugin v1.17.13` 类型定义和官方文档逐点校验 |
| 结论 | 设计方向正确，但代码中多个核心 API 调用基于虚构字段，**不可直接使用** |

---

## 一、API 事实核查

### 1.1 shell.env 钩子

| 维度 | 官方 API | 方案代码 | 差异 |
|------|----------|----------|------|
| output 结构 | `{ env: Record<string, string> }` | `envCtx.preCommand` + `envCtx.vars` | ❌ 两个字段均不存在 |
| 注入方式 | `output.env.KEY = "value"` | `envCtx.vars = { ... }` | ❌ 写法全错 |
| 能力边界 | 只能设环境变量 | 企图注入命令前缀 + 批量变量 | ❌ 超出能力 |

**判定：不可行。** `preCommand` 字段在 OpenCode 插件 API 中完全不存在。`shell.env` 只能通过 `output.env.KEY = "VALUE"` 逐一设置环境变量。

### 1.2 tool.execute.before 钩子

| 维度 | 官方 API | 方案代码 | 差异 |
|------|----------|----------|------|
| input 类型 | `{ tool, sessionID, callID }` | `input.tool` | ✅ 使用正确 |
| output 类型 | `{ args: any }` | `ctx.args.command` | ✅ 对 bash 工具可行 |
| 参数命名 | `output` | `ctx` | ⚠️ 有误导但功能不影响 |

**判定：逻辑上可行。** 通过 `output.args.command` 改写命令字符串是正确用法。

## 二、致命缺陷（按严重程度排列）

### 🔴 致命 1 — preCommand 是虚构能力

方案的**核心卖点**——"每条命令自动前置 UTF8 编码脚本 + 加载 `$PROFILE`"——完全依赖 `envCtx.preCommand` 字段，该字段在 OpenCode 中**不存在**。

`shell.env` 只能注入环境变量，不能拼接命令。编码设置（`[Console]::OutputEncoding=...`）是 PowerShell 命令，不可能通过此钩子实现。

**影响**：模块 A 的编码前置部分完全不可行。整个方案的"一次配置永久生效"承诺无法实现。

### 🟡 严重 2 — . $PROFILE 自动加载是有害设计

即使有办法拼接，对每条命令 dot-source `$PROFILE` 有严重副作用：

| 风险 | 说明 |
|------|------|
| **性能** | PROFILE 若加载 oh-my-posh、posh-git、PSReadLine 等模块，每条命令增加数百毫秒到数秒 |
| **输出污染** | PROFILE 中的 Prompt 函数、Write-Host、ANSI 颜色设置会污染 stdout，导致 OpenCode 解析出错 |
| **兼容性** | PROFILE 设计给交互式 session 用，非交互子进程中许多命令会报错甚至中断执行 |

**建议**：改用精确的编码配置命令替代 `$PROFILE` 加载。

### 🟡 严重 3 — 正则表达式有多个边界漏洞

```regex
/^\$?export\s+((?:\w+=[^; ]+\s*)+)(;.*)?/s
```

| 问题 | 示例 | 错误行为 |
|------|------|----------|
| `\$?export` 匹配 `$export` | `$export FOO=bar` | bash 中无此命令，永远不应匹配 |
| `[^; ]+` 不处理引号 | `export FOO="hello world"` | 空格处断开，生成 `$env:FOO=""hello` |
| 只捕获 `;` 后内容 | `export FOO=bar && make` | `&& make` 丢失 |
| 按空格 split 忽略引号 | `export A="x y" B=z` | 错误地将 `"x` 当值，`y"` 当另一个 key |
| 不处理多行续行 | `export FOO=bar \` + 换行 + `BAZ=qux` | 完全失效 |

### 🟠 中等 4 — 环境变量无法阻止 Agent 生成 export

方案设想"预注入 CI 环境变量减少 Agent 生成 export"，但 Agent 生成 `export FOO=bar` 是因为：

- 训练数据中 bash 语法占绝大多数
- 模型按惯性输出 Linux 风格指令

注入环境变量**不会改变 LLM 的输出行为**。真正有效的方法是在 Agent 的 system prompt 中明确声明（你的 `oh-my-openagent.jsonc` 已在 Sisyphus 的 instructions 中做了这件事）。

## 三、方案中可保留的部分

| 部分 | 可行性 | 需要修正 |
|------|--------|----------|
| 双钩子拆分设计 | ✅ 正确 | API 签名需按真实接口重写 |
| `tool.execute.before` 拦截 export | ✅ 可行 | 需重写正则表达式 |
| `shell.env` 注入 CI 环境变量 | ✅ 可行 | 改用 `output.env.KEY = "VALUE"` 逐一赋值 |
| 编码前缀注入 | ✅ 可行 | 移至 `tool.execute.before` 通过 `output.args.command` 拼接 |
| 去除 `$PROFILE` 依赖 | ✅ 正确决策 | 替换为精确的 `$OutputEncoding` + `[Console]::OutputEncoding` |

## 四、修复路径

完整的修正方案见 `02-完整修复方案.md`。核心变更：

1. **`shell.env` 只设环境变量**：`output.env.CI = "true"` 逐一注入
2. **编码前置移到 `tool.execute.before`**：通过 `output.args.command` 拼接
3. **去掉 `. $PROFILE`**：改为只设两条编码命令
4. **重写正则**：支持带引号值、多对 K=V、保留剩余命令
5. **补上 PowerShell 转义**：正确处理 `$`、`"`、`` ` `` 字符
